# sqlmodel-nexus 代码审查报告

## 一、代码质量与最佳实践问题

### 1. ~~Resolver 缺少元数据缓存（性能问题）~~ [DONE]

> **已修复**: 添加了 `_ClassMeta` 数据类和模块级 `_class_meta_cache`。`dir()` + `inspect.signature()` 每个类只执行一次，后续从缓存读取。

### 2. ~~`_prepare_collectors` 每次都重新解析方法签名~~ [DONE]

> **已修复**: `_prepare_collectors` 改为使用 `_ClassMeta.collector_aliases` 中预计算的 Collector 信息，不再重复扫描方法签名。

### 3. ~~`_node_collectors` 使用 `id(node)` 作为 key 存在内存泄漏风险~~ [DONE]

> **已修复**: 在 `_traverse()` 的 `finally` 块中添加 `self._node_collectors.pop(id(node), None)`，节点遍历完成后立即释放。

### 4. ~~`scan_expose_fields` / `scan_send_to_fields` 缺少缓存~~ [DONE]

> **已修复**: 添加了模块级 `_expose_cache` 和 `_send_to_cache`，扫描结果按类缓存。

### 5. `subset.py` 中的 `eval()` 使用

**文件**: `subset.py:286, 503, 530`

```python
resolved = eval(resolved, _global_ns, namespace)  # noqa: S307
```

虽然有 `noqa` 标记，但这是处理 `from __future__ import annotations` 字符串注解的方式。实际上 Python 3.10+ 的 `typing.get_type_hints()` 可以更安全地解析。如果用户的类定义中有恶意注解（尽管不太可能），`eval` 可能执行任意代码。

### 6. ~~`_extract_sort_field` 假设 `order_by` 只有 `.key` 属性~~ [DONE]

> **已修复**: 添加了对 `desc(Column)` / `asc(Column)` (`UnaryExpression`) 的处理，通过 `order_by.element.key` 提取列名。

### 7. ~~`get_loader_by_name` 全局搜索可能返回错误的 loader~~ [DONE]

> **已修复**: 改为先收集所有匹配项，当存在多个同名 relationship 时记录 warning 日志，提示使用 `get_loader_for_entity()` 替代。

### 8. ~~`query_meta.py` 中 FK 字段推导假设 `{rel_name}_id` 命名~~ [DONE]

> **已修复**: `generate_query_meta_from_selection()` 和 `generate_type_key_from_selection()` 新增 `fk_lookup` 参数，由 `QueryExecutor` 从 `LoaderRegistry` 构建实际 FK 映射传入。`{rel}_id` 仅作为 fallback。

### 9. ~~DataLoader factory 将类属性直接设置在动态类上~~ [DONE]

> **已修复**: 所有 5 个 factory 函数改为闭包捕获模式，`batch_load_fn` 直接引用外层作用域变量，不再设置类属性。

### 10. `QueryExecutor._results` 使用 `id(entity)` 做 key

**文件**: `execution/query_executor.py:43-51`

```python
self._results: dict[tuple[int, str], Any] = {}

def _store(self, entity: Any, field_name: str, value: Any) -> None:
    self._results[(id(entity), field_name)] = value
```

同 `_node_collectors` 一样，`id()` 在对象生命周期结束后可复用。在单次请求内通常安全，但如果有对象被 GC 后新对象获得相同 id，可能导致数据错位。

### 11. `handler.py` 的 `execute()` 捕获所有异常

**文件**: `handler.py:154-173`

```python
async def execute(self, query, ...):
    try:
        ...
    except Exception as e:
        logger.exception("GraphQL execution error")
        return {"errors": [{"message": str(e)}]}
```

捕获所有 `Exception` 包括 `KeyboardInterrupt` 以外的所有异常，可能掩盖严重的编程错误（如 `AttributeError`、`TypeError`）。GraphQL spec 允许 partial data + errors，但不应该吞掉内部 bug。建议至少对 `ValueError`/`TypeError` 等分开处理。

### 12. Pagination loader 中的 `has_more` 判断逻辑

**文件**: `loader/factories.py:443-446`

```python
has_next_page=total_counts.get(cmd.fk_value, 0) >= end
if cmd.fk_value in total_counts
else False,
```

`has_next_page` 判断条件 `total_count >= end` 实际上应该是 `total_count > end - 1`，即 `total_count >= start + effective_limit`。当 `offset=0, limit=10, total=10` 时，`end=11`，`10 >= 11` 为 False，正确。但如果 `total=11`，`11 >= 11` 为 True，也正确。仔细看逻辑是对的，但表达式可读性差，应该用更清晰的方式。

### 13. `_resolve_relationships` 中跳过无 sub_fields 的 relationship

**文件**: `execution/query_executor.py:173-179`

```python
for field_name, child_sel in field_sel.sub_fields.items():
    if not child_sel.sub_fields:
        rel_info = self._registry.get_relationship(parent_entity, field_name)
        if rel_info is None:
            continue
        continue  # ← 即使是 relationship 也跳过了
```

如果用户查询 `{ users { posts } }` 而没有选择 posts 的子字段，relationship 会被完全跳过。这可能不符合用户预期——他们可能期望至少返回空列表或默认序列化。

## 二、测试覆盖度分析

### 覆盖良好的领域
- `DefineSubset` / `SubsetConfig` - ~50 个测试，覆盖各种配置
- `Resolver` 基本功能 - resolve_*/post_* 执行、上下文传递
- `LoaderRegistry` 发现与注册
- GraphQL SDL 生成
- Introspection 支持
- MCP 服务器（单应用、多应用、工具注册）
- AutoLoad 机制（显式与隐式）
- ExposeAs / SendTo / Collector 数据流

### 关键缺失的测试

| 缺失测试 | 影响 | 建议 |
|----------|------|------|
| **Pagination 端到端** | 分页的 ROW_NUMBER + limit/offset + has_more 在真实数据库中未被测试 | 使用 conftest 的 fixture 数据测试分页查询 |
| **M2M relationship 执行** | `create_many_to_many_loader` 和 `create_page_many_to_many_loader` 未在真实数据库场景中测试 | 添加 M2M fixture 测试 |
| **并发安全** | 多个 `Resolver` 实例同时运行（共享 `contextvars`）是否安全 | 添加 `asyncio.gather` 多 resolver 并发测试 |
| **大数据量** | 1000+ 个节点的性能和正确性 | 至少一个大数据集测试 |
| **`_query_meta` / column pruning** | `_get_effective_query_fields` + `_apply_load_only` 的实际 SQL 效果 | 验证生成的 SQL 确实只选了需要的列 |
| **`_orm_to_dto` 异常路径** | `resolver.py:333-348` 的 `model_rebuild` fallback 路径 | 添加需要 rebuild 的场景测试 |
| **变量传递 (GraphQL variables)** | `execute_query` 接受 variables 但无测试 | 添加带 `$variable` 的 GraphQL 查询测试 |
| **Enum 类型在实际执行中** | SDL 生成了 Enum 但执行时传 Enum 参数的路径未测试 | 添加 Enum 参数的查询/变更测试 |
| **错误恢复** | 一个 resolve_* 抛异常后，其他同级 resolve 是否正常完成 | 测试 partial failure 场景 |

## 四、总结与建议

### 高优先级

1. ~~**添加 Resolver 元数据缓存**~~ [DONE] — 已添加 `_ClassMeta` + `_class_meta_cache`，`dir()` + `inspect.signature()` 每类只执行一次。
2. ~~**修复 `query_meta.py` 的 FK 命名硬编码**~~ [DONE] — 已通过 `fk_lookup` 参数从 `LoaderRegistry` 获取实际 FK 字段名。
3. **补充 Pagination 端到端测试** — 分页是重要功能但缺少真实数据库测试。
4. **添加自定义异常类** — 当前错误信息不够结构化，调试困难。

### 中优先级

5. **迁移 `mapper()` 装饰器** — 对 resolve_* 结果做后处理是常见需求。
6. **导出 `build_list()` / `build_object()`** — 用户编写自定义 loader 时需要这些辅助函数。
7. **实现 DataLoader `_context` 注入** — 让自定义 loader 可以访问 Resolver 的 context。
8. **迁移 `Relationship.fk_fn` / `fk_none_default`** — 处理 FK 值不是简单直接映射的场景。

### 低优先级（可根据需要迁移）

9. `loader_params` / `global_loader_param` — 按需配置 loader 的能力
10. `debug` / Profile — 性能分析
11. `ensure_type` — 开发阶段的类型安全检查
12. `copy_dataloader_kls` — 便捷工具函数

整体来看，项目结构清晰，核心功能（GraphQL + Core API 双模式、AutoLoad、DefineSubset、ExposeAs/SendTo/Collector 数据流、MCP 集成）实现完整。主要差距在**开发者工具**（异常、调试、mapper、辅助函数）方面。性能优化（缓存）已完成。
